THE PROBLEM
Every product company runs a notification service
Razorpay, Swiggy, Groww, PhonePe - they all maintain internal notification infrastructure. It solves the same real infrastructure problem: reliable, multi-channel notification delivery with queue-based architecture, retry logic, dead letter queues, idempotency and full observability.
Core Architecture
▶
BullMQ Queue System
Dedicated queues per channel backed by Redis. Jobs are enqueued after DB insert and processed by isolated workers.
↻
Exponential Backoff Retry
Failed jobs retry with a calculated delay. After 3 attempts, jobs move to dead letter queue (DLQ) for manual inspection.
❖
Idempotency Keys
DB-level unique constraints and middleware enforce exactly-once delivery. Duplicate requests return 409.
◉
Prometheus + Grafana
Live dashboards showing queue depth, delivery latency, error rates, notifications. Correlation IDs trace every request.
